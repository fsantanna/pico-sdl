# `_pico_guard()` audit

Which public `pico_*` functions don't need the init guard?

Two groups:
**pure math (no `G` access)** and the **init function itself**.

| function              | uses `G`? | currently guards? | needs guard? |
|-----------------------|-----------|-------------------|--------------|
| `pico_color_darker`   | no        | no                | **no**       |
| `pico_color_lighter`  | no        | no                | **no**       |
| `pico_color_mix`      | no        | no                | **no**       |
| `pico_color_alpha`    | no        | no                | **no**       |
| `pico_color_hex`      | no        | no                | **no**       |
| `pico_in_rect`        | no        | no                | **no**       |
| `pico_in_pos`         | no        | no                | **no**       |
| `pico_in_dim`         | no        | no                | **no**       |
| `pico_init`           | yes       | no                | **no** (manages init state) |

Everything else (`pico_quit`, all `set_*`, `get_*`, `output_*`,
`layer_*`, `input_*`, `cv_*`, `vs_*`) touches `G` and **does** need
the guard.

## Why these are guard-free

**`pico_color_*`** (colors.hc:74-121):
pure RGBA arithmetic on the input value, returns a new color.
No SDL state, no global access.

**`pico_in_*`** (geom.hc:209-231):
compose rect/pos/dim onto a parent rect via `_sdl_rect` / `_rel_rect`
helpers, **always passing `NULL` as the scene base**
(`_sdl_rect(out, NULL, NULL)`).
The helpers don't touch `G` when scene base is NULL — they just do
math on the inputs.

So these 8 utility functions are usable *before* `pico_init(true)` is
called, which is a nice property (you can pre-compute colors and
compose rects at static init time).
