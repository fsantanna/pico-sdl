# Remove `up` from Pico_Rel_{Rect,Dim,Pos}

## Context

The three relative-coord structs in `src/pico.h` carry a
`struct Pico_Rel_Rect* up` pointer that builds a reference-frame chain.
At resolution time, `_f1` in `src/aux.hc` walks the chain recursively to
compose transforms.

The user wants the chain mechanism gone.
The structs flatten to plain coordinate triples.
Migration policy:

- `.up = NULL` literals (and asserts on `up==NULL`):
    - **delete silently**, no behavior change once the field is gone
- `.up = <non-NULL expr>` literals:
    - **leave the assignment in place**
    - the compiler errors at every callsite that genuinely propagated a
      parent rect
    - the user reworks each one by hand

## `_f1` chain walker — collapse, do not delete

`src/aux.hc:33-45`.
`_f1` is also used as a single-frame transform helper (see `_sdl_rect`
`'%'` branch at L240: `_f1(rect, r0, ratio)`).
Drop only the recursion:

```c
static SDL_FRect _f1 (
    const Pico_Rel_Rect* r,
    SDL_FRect base,
    const Pico_Abs_Dim* ratio
) {
    if (r == NULL) {
        return base;
    } else {
        SDL_FRect abs = { r->x, r->y, r->w, r->h };
        return _f2(abs, r->anchor, base, ratio);
        // was: SDL_FRect tmp = _f1(r->up, base, NULL);
        //      return _f2(abs, r->anchor, tmp, ratio);
    }
}
```

Every caller that previously passed `dim->up` / `pos->up` / `rect->up`
to `_f1` no longer has a chain to walk; reduce to `r0` directly:

| src/aux.hc line | original                                                                        | becomes              |
| 122             | `SDL_FRect r1 = _f1(dim->up, r0, NULL);`                                        | `SDL_FRect r1 = r0;` |
| 168             | `SDL_FRect r1 = _f1(pos->up, r0, NULL);`                                        | `SDL_FRect r1 = r0;` |
| 218             | `SDL_FRect r1 = _f1(rect->up, r0, NULL);`                                       | `SDL_FRect r1 = r0;` |
| 277             | `SDL_FRect r1 = _f1(to->up, r0, NULL);`                                         | `SDL_FRect r1 = r0;` |
| 313-314         | `Pico_Rel_Rect* up = (Pico_Rel_Rect*) to->up; SDL_FRect r1 = _f1(up, r0, NULL);` | `SDL_FRect r1 = r0;` |
| 350             | `SDL_FRect r1 = _f1(to->up, r0, NULL);`                                         | `SDL_FRect r1 = r0;` |

The self-call `_f1(rect, r0, ratio)` at line 240 (in `_sdl_rect '%'`)
stays — it uses `_f1` as a one-shot transform, not chain traversal.

### Templates `_C_tpl_*` (lines 432-491)

- Delete `_C_tpl_up` entirely (432-442).
- In `_C_tpl_dim` (444), `_C_tpl_pos` (459), `_C_tpl_rect` (476):
    - remove `Pico_Rel_Rect* up = _C_tpl_up(L, i);`

the key ignored.
That is the intended quiet-removal half.
Follow-up if you want loud failure:
emit a `luaL_error` when an `up` key is seen.

## Open design questions for the KEEP sites

The six KEEP sites all share one pattern:
they construct a `Pico_Rel_Dim` (or `Pico_Rel_Pos`) by copying
`mode/w/h` from a `Pico_Rel_Rect*` and pass the parent's reference
frame via `up`.
With `up` gone, the parent rect must travel via a new explicit
parameter — likely the existing `base: Pico_Abs_Rect*` argument on the
`_sdl_*` helpers, or a new `Pico_Rel_Rect*` parameter on the public
API.

These six are deliberately left as compile errors so the user makes
the design call site by site before the plan moves to a "compiles
clean" finish.

### Deferred design decision: `src/mem.hc:178` (sub-layer src chain)

Original line (now commented out):
```c
data->scene.src.up = &c->par->scene.src;
```

It chained the sub-layer's `scene.src` onto the parent's `scene.src`,
so resolution at draw time composed both — parent scrolling/zooming
would drag sub-layers along.

With `.up` gone, three options (deferred until intended semantics
confirmed):

| option | behavior | when right |
| (a) **delete the line** | sub-layer's src is `c->crop`, never tracks parent | sub-layers are static crops |
| (b) **compose at alloc time** — `data->scene.src = compose(c->crop, c->par->scene.src);` | freezes parent's src-at-creation into the crop | parent's src is fixed at `pico_layer_sub` time |
| (c) **resolve on demand via parent ptr** — extend `Pico_Layer_Sub.sup` (already holds parent's `scene.dim`) to also carry parent's `scene.src` (or store parent layer pointer); compose at draw | preserves old chain semantics | parent scrolling should drag sub-layers |

Open question for the user:
should sub-layers follow parent `scene.src` changes after creation?
