# 2026-05-22 push-pop

Expose realm scopes to user code via `pico_push` / `pico_pop`, so a
group of layers and resources can be released together when a scene
ends (typical case: menu → game transition).

## Goal

Today the realm is opened once in `pico_init(true)` and closed once
in `pico_init(false)` (`src/pico.c:81,215`). The internal
`realm_enter` / `realm_leave` (`src/realm.hc:103-120`) already
support stacked scopes via `depth`, but only one `realm_enter`
runs (`src/pico.c:84`) — there is no way for user code to bracket a
scene.

Expose two verbs:

```
pico_push (void);   // begin scope
pico_pop  (void);   // end scope — frees every entry created since
                    //              the matching push
```

No `pop(key)`. Per-key replacement keeps using realm mode `'~'`,
which already does free-old-then-alloc-new on the same entry.

## Use case (Lua)

```lua
-- menu
pico.push()
pico.layer.image("world", "bg",   "menu_bg.png")
pico.layer.text ("world", "play", 40, "PLAY", ...)
-- ... menu loop ...
pico.pop()       -- frees bg, play, every layer created in this scope

-- game
pico.push()
pico.layer.image("world", "lvl1", "lvl1.png")
pico.layer.pixmap("world", "hero", ...)
-- ... game loop ...
pico.pop()
```

Scene-replace idiom is just `pop()` followed by `push()`.

## Pre-requisite: detach-on-free

Each `Pico_Layer.name` is `strdup`'d and freed by
`_pico_mem_free_layer`. Three other places cache borrowed pointers
to it:

| ref                         | held by         |
|-----------------------------|-----------------|
| `hier.dn.fst` / `hier.dn.lst` | parent          |
| `hier.nxt`                   | previous sibling |
| `hier.up`                    | each child      |

Without cleanup, any of the three free paths (`'~'` replace,
`realm_leave` pop, `realm_close` shutdown) leaves dangling
pointers — the next chain walk crashes at
`assert(CUR != NULL)` (`src/layer.c:39`).

Fix: `_detach_layer` (static in `src/mem.c`), called at the top
of `_pico_mem_free_layer`. Two blocks:

1. **Splice from parent**: if `self->hier.up` resolves to a live
   `UP`, walk `UP->hier.dn.fst` chain, find self, rewrite the
   predecessor's `nxt` (or `UP->hier.dn.fst`) to skip self.
   Update `UP->hier.dn.lst` if self was the tail.

2. **Disown own children**: walk `self->hier.dn.fst` chain, set
   each `CUR->hier.up = NULL` so children that outlive us
   (`realm_close` bucket order, same-depth pop) don't read the
   freed `self->name`.

Both blocks assert chain invariants (parent resolvable, chain
links live, self found, back-pointer matches, tail matches
`hier.dn.lst`).

### Static layers (`G.world`, `G.window.layer`)

These are not realm-owned (no strdup, no free) but ARE in the
realm so `realm_get("world")` resolves. Without a free callback,
`realm_close` removes their entries silently, leaving children
with `hier.up = "world"` (literal, still valid) but no realm
entry — `realm_get` returns NULL and the first-block assert
fires. Fix: `_pico_mem_detach_layer` — a realm callback that
runs `_detach_layer` only, leaving the static struct, texture,
and string-literal name untouched (`pico_init(false)` handles
those).

### `'~'` replace on a parent — error

`_detach_layer` keeps `'~'` replace memory-safe, but it silently
orphans the children of the replaced layer (their `hier.up` is
nulled; they remain in the realm but no parent renders them).
Caught early with a pre-assert in every `pico_layer_*_mode`:

```c
if (mode == '~') {
    Pico_Layer* old = realm_get(G.realm, strlen(key)+1, key);
    if (old != NULL) {
        assert(old->hier.dn.fst == NULL
            && "cannot '~' replace a layer with children");
    }
}
```

To swap a subtree, user explicitly tears down children first.

## C API

```c
// src/pico.h
void pico_push (void);
void pico_pop  (void);
```

```c
// src/pico.c
void pico_push (void) {
    _pico_guard();
    realm_enter(G.realm);
}

void pico_pop (void) {
    _pico_guard();
    pico_assert(G.realm->depth > 1 && "pop without matching push");
    realm_leave(G.realm);
}
```

`depth > 1` guard: the init scope (`window`, `world`) lives at
depth 0 with `r->depth == 1` after the one-time `realm_enter`. A
user `pop` must never drop below that floor.

`pico_init(false)` is unchanged: `realm_close` walks every bucket
regardless of depth, so a user who forgot a `pop` still gets a
clean shutdown.

## Lua binding

```c
// lua/pico.c
static int l_push (lua_State* L) { pico_push(); return 0; }
static int l_pop  (lua_State* L) { pico_pop();  return 0; }

// register: pico.push, pico.pop
```

No sniffing, no varargs — zero-arg verbs.

Optional sugar in `lua/pico/init.lua`:

```lua
function pico.with (fn)
    pico.push()
    local ok, err = pcall(fn)
    pico.pop()
    if not ok then error(err, 2) end
end
```

`pico.with(function() ... end)` guarantees a `pop` even on error.
Lands as follow-up if useful; not required for v1.

## Realm modes

Keep all three modes (`'!'`, `'='`, `'~'`). They solve a different
problem (insertion-conflict resolution per key) from scope
lifetime. `'~'` is *not* redundant with `pop+push`:

- `pop+push` clears a whole scope.
- `'~'` replaces a single key in place — useful mid-scene without
  tearing down the surrounding scope.

`'~'` replace neutralises one of the three dangling paths

## Tradeoff considered

| approach                | pro                              | con                            |
|-------------------------|----------------------------------|--------------------------------|
| `push`/`pop` only       | minimal API, matches realm core  | no per-key delete              |
| `push`/`pop` + `pop(k)` | granular cleanup                 | overlaps with `'~'`, more API  |
| Refcounted entries      | safe `pop(k)` for shared (`'='`) | adds a counter to every entry  |

**Pick:** `push`/`pop` only. Per-key needs are served by `'~'`,
and shared (`'='`) keys are intentionally long-lived — letting
them die with the scope is fine.

## Resolved

- `pico_pop` asserts `G.layer == &G.world` — user must restore the
  target to `world` before popping. No snap-back, no resolve-by-name.
- No warning when `pop` frees `'='`-shared entries. Caller's
  responsibility.
